PEC ERP FACULTY APP - NEXT.JS TO FLUTTER IMPLEMENTATION PLAN
Location: mobile/Faculty_App
Date: 2026-04-08

OBJECTIVE
- Build the Flutter Faculty_App UI to match the existing Next.js faculty web frontend as closely as possible.
- Keep one common backend (NestJS) and one common database (PostgreSQL) for web + mobile.
- Keep one common authentication/session model for Next.js and Flutter clients.
- Ensure mobile frontend is responsive across phone and tablet form factors.

NON-NEGOTIABLE ARCHITECTURE RULES
1. Do not create a separate backend for Faculty_App.
2. Do not connect Flutter directly to PostgreSQL.
3. Keep NestJS APIs shared across Next.js, pec_app, and Faculty_App.
4. Keep role-based authorization in backend guards, not only in UI.
5. Any data contract change must be backward-compatible or versioned.

SYSTEM ARCHITECTURE (FINAL TARGET)
- Client 1: Next.js website (already implemented for faculty workflows).
- Client 2: Flutter Faculty_App.
- Client 3: Flutter pec_app.
- Server: Common NestJS API layer in server/.
- DB: Common PostgreSQL via Prisma schema in server/prisma/schema.prisma.

API + AUTH + DB COMMONALITY REQUIREMENTS
- Same auth endpoints for all clients: login, refresh, profile, logout.
- Same JWT/refresh lifecycle for web and Flutter.
- Same role mapping (faculty, admin, student, etc.).
- Same API routes and payload semantics.
- Same data source of truth (PostgreSQL).

RESPONSIVE REQUIREMENTS (MANDATORY)
- Mobile breakpoint strategy:
  Compact phones: <= 360 width
  Standard phones: 361-600 width
  Tablet portrait: 601-900 width
  Tablet landscape/desktop-webview: > 900 width
- Layout behavior:
  1-column on compact phones
  2-column adaptive cards on larger phones/tablets
  Split panes for selected screens on tablets where meaningful
- Typography and spacing:
  Use scaled text styles with clamped min/max sizes
  Use adaptive paddings and gutters
- Inputs and accessibility:
  Touch targets >= 44dp
  High contrast and readable text hierarchy

API PARITY MATRIX (USE SAME ROUTES)
Dashboard data routes used by web:
- GET /courses?facultyId={uid}&limit=20
- GET /attendance/faculty-stats
- GET /auth/profile
- GET /noticeboard?limit=5

Faculty Bio System routes used by web:
- GET /faculty-bio-system/{facultyId}

Publication CRUD:
- POST /faculty-bio-system/publications
- PATCH /faculty-bio-system/publications/{id}
- DELETE /faculty-bio-system/publications/{id}

Award CRUD:
- POST /faculty-bio-system/awards
- PATCH /faculty-bio-system/awards/{id}
- DELETE /faculty-bio-system/awards/{id}

Conference CRUD:
- POST /faculty-bio-system/conferences
- PATCH /faculty-bio-system/conferences/{id}
- DELETE /faculty-bio-system/conferences/{id}

Consultation CRUD:
- POST /faculty-bio-system/consultations
- PATCH /faculty-bio-system/consultations/{id}
- DELETE /faculty-bio-system/consultations/{id}

IMPORTANT
- Reuse these exact endpoints for Flutter.
- Do not introduce duplicate /mobile/* routes.
